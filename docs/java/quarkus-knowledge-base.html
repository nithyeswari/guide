01 Origins & History

Project Shamrock (2018)

Quarkus began as an internal Red Hat skunkworks project codenamed "Project Shamrock" in mid-2018. The core team at Red Hat observed that the Java ecosystem was losing ground in cloud-native deployments to Go, Node.js, and Rust — primarily because of JVM startup time, memory footprint, and container image size. The mission: build a framework that makes Java first-class in Kubernetes and serverless environments.

Core Team

The project was led by Emmanuel Bernard (Hibernate co-lead), Jason Greene (WildFly lead), and Bob McWhirter (Thorntail/WildFly Swarm). They assembled engineers from the Hibernate, RESTEasy, Vert.x, WildFly, and MicroProfile teams. The key insight was to move framework initialization work — classpath scanning, annotation processing, proxy generation, config parsing — from runtime to build time.

The Breakthrough

Rather than trying to optimize existing frameworks, the team built a new build-time augmentation architecture from scratch. This "closed world assumption" means Quarkus assumes the application's class graph is fully known at build time. This single design decision enabled sub-second startup, drastically lower memory consumption, and seamless GraalVM native image compilation — all without sacrificing the developer experience Java programmers expect.

02 The Quarkus 0.11 Launch

The Java Cloud Problem

By 2018–2019, the cloud landscape had shifted decisively toward containers and serverless. Traditional Java applications suffered from three critical weaknesses in this environment:

  • Startup time: Spring Boot applications typically took 5–30 seconds to start, making them impractical for serverless cold starts and rapid Kubernetes scaling.
  • Memory footprint: A simple "hello world" REST service in Spring Boot consumed 200–400 MB of RSS memory — 10x more than equivalent Go or Node.js services.
  • Container image size: Fat JARs + JDK base images = 300–700 MB images, slowing pull times and increasing registry costs.

Quarkus 0.11 (March 2019) was the first public demonstration that Java could compete on these dimensions without abandoning the ecosystem.

Key Features at Launch

Build-Time Augmentation

Framework bootstrap logic executed at build time via the augmentation pipeline. Classes, proxies, and configuration metadata are pre-computed and baked into the output artifact.

GraalVM Native Image

First-class support for compiling Java applications to standalone native executables. No JVM required at runtime. Sub-20ms startup.

Live Coding (Dev Mode)

mvn quarkus:dev enables hot-reload on every HTTP request. Code changes are reflected instantly without restarting the JVM.

Standards-Based

Built on CDI, JAX-RS, JPA, Bean Validation, MicroProfile — not proprietary APIs. Developers use standard annotations and patterns.

Unified Imperative & Reactive

Both @Blocking and reactive (Mutiny Uni/Multi) programming models coexist in the same application on the same Vert.x event loop.

Extension Ecosystem

700+ extensions at launch provided turnkey integrations for databases, messaging, cloud services, security, and observability.

Release Cadence

Since launch, Quarkus has maintained a rapid release cadence:

  • Minor releases: every 4–6 weeks (e.g., 3.14 → 3.15 → 3.16)
  • Patch releases: as needed for security and critical fixes
  • LTS releases: approximately every 12 months (3.2 LTS, 3.8 LTS, 3.20 LTS)
  • Major releases: every 2–3 years for breaking changes (1.0, 2.0, 3.0, 4.0)
03 Maven Build System

Quarkus Packaging Type

Quarkus uses the standard Maven jar packaging type but introduces a custom build lifecycle via the quarkus-maven-plugin. The plugin hooks into Maven's package phase to run the Quarkus augmentation pipeline, producing optimized build outputs (fast-jar, uber-jar, or native binary).

Plugin Goals

Goal Phase Description
quarkus:dev N/A (interactive) Starts dev mode with live reload, continuous testing, and dev services
quarkus:test N/A (interactive) Runs continuous testing in isolation (no dev mode)
quarkus:build package Runs augmentation and produces the configured package type
quarkus:generate-code generate-sources Runs code generation (gRPC stubs, jOOQ, GraphQL schema, etc.)
quarkus:generate-code-tests generate-test-sources Generates test-specific code (test gRPC clients, etc.)
quarkus:add-extension N/A Adds an extension dependency to pom.xml interactively
quarkus:list-extensions N/A Lists available extensions and their installation status
quarkus:image-build package Builds a container image (Docker/Podman) using Jib, Docker, or buildpack
quarkus:image-push N/A Pushes the built container image to a registry

Build Lifecycle

compile Standard javac compilation (no Quarkus involvement) .class files in target/classes

test @QuarkusTest boots a full Quarkus app for integration tests Test results

Fast-Jar Directory Structure

The fast-jar (default since Quarkus 1.12) is a directory-based packaging format optimized for startup speed. Dependencies are indexed and loaded via a custom classloader that avoids the overhead of scanning a single uber-jar.

Directory target/quarkus-app/ 
target/quarkus-app/
  quarkus-run.jar            # Entry point (small bootstrap JAR, ~500 KB)
  app/
    my-app-1.0.jar            # Your application classes
  lib/
    boot/
      *.jar                    # Bootstrap classloader JARs
    main/
      *.jar                    # Runtime dependencies (pre-indexed)
  quarkus/
    generated-bytecode.jar   # Build-time generated classes (recorders, proxies)
    quarkus-application.dat  # Serialized application metadata
    transformed-bytecode.jar # Bytecode-transformed classes (Panache, etc.)

05 How Quarkus Is Fundamentally Different

Paradigm Shift

Quarkus represents a fundamental paradigm shift in how Java frameworks operate. Traditional frameworks (Spring Boot, Jakarta EE app servers) perform massive amounts of work at runtime startup. Quarkus moves this work to build time.

Concern Traditional (Runtime) Quarkus (Build Time)
Classpath Scanning At every startup — scans all JARs for annotations Once at build — results serialized into metadata
CDI Bean Discovery Runtime reflection on every class Build-time Jandex index; bean graph pre-resolved
Proxy Generation Runtime bytecode generation (CGLIB, ByteBuddy) Pre-generated at build; loaded as static classes
Configuration Parsing Parsed and validated at startup Parsed at build; only runtime values deferred
ORM Metadata Entity scanning and metadata building at startup Entity model pre-built; Hibernate metamodel baked in
Reflection Registration Implicit — any class can be reflected Explicit — only declared reflective targets kept
Dead Code Loaded anyway — JVM doesn't know what's unused Eliminated — unreachable extensions/classes removed

Traditional Model (Runtime)

Traditional Framework Startup
Application Start | v [Classpath Scan] ---- 1-3 seconds (scan every JAR for annotations) | v [Bean Discovery] ---- 0.5-2 seconds (reflect on thousands of classes) | v [Proxy Generation] --- 0.5-1 second (generate CGLIB/ByteBuddy proxies) | v [Config Parsing] ---- 0.2-0.5 seconds (parse YAML/properties, validate) | v [ORM Bootstrap] ---- 1-3 seconds (entity scan, schema validation) | v [Server Start] ---- 0.3-0.5 seconds | v READY --- Total: 4-10 seconds

Quarkus Model (Build Time)

Quarkus Build-Time Augmentation
BUILD TIME (mvn package) RUNTIME (java -jar) ======================== ==================== [Jandex Indexing] [Load pre-built metadata] | | [Bean Graph Resolution] [Execute recorded bytecode] | | [Proxy Generation] [Start Vert.x/Netty] | | [Config Defaults Baked] [Open listening socket] | | [ORM Metamodel Built] READY | | [Recorder Bytecode] Total: 0.5-1.0 seconds | [Dead Code Elimination] | [Serialize to .dat file] | Total: 10-30 seconds (once)

The Closed World Assumption

🔒
Closed World Assumption: Quarkus assumes that the set of classes and resources in the application is fully known at build time and will not change at runtime. This means:

• No runtime class loading of new JARs
• No dynamic proxy generation at startup
• No runtime annotation scanning
• No dynamic classpath modifications

This assumption is what makes GraalVM native image compilation feasible and is the single most important architectural decision in Quarkus. It trades dynamic flexibility (rarely used in production microservices) for dramatically improved startup, memory, and image size.
06 Augmentation Deep-Dive

5-Stage Pipeline

The Quarkus augmentation pipeline runs at build time (during mvn package) and consists of five sequential stages. Each stage processes the application and produces artifacts consumed by subsequent stages.

1
Index
Jandex scans all application and dependency classes. Produces an in-memory annotation index — no reflection needed at runtime.
→
2
Augment
Extension build steps (@BuildStep) run against the index. Each extension contributes BuildItems — beans, resources, config defaults, reflection registrations.
→
3
Record
@Recorder methods capture runtime initialization logic as bytecode. These are serialized sequences of method calls replayed at startup.
→
4
Transform
Bytecode transformations applied: Panache entity enhancement, CDI proxy generation, REST endpoint optimization. Output: transformed .class files.
→
5
Package
All artifacts assembled into the chosen format (fast-jar, uber-jar, or native). Metadata serialized to quarkus-application.dat.

The @Recorder Mechanism

Recorders are the bridge between build time and runtime. A @Recorder class defines methods that will be invoked at runtime startup, but the invocation sequence is determined at build time.

Java DatasourceRecorder.java 
@Recorder
public class DatasourceRecorder {

    // This method is called at RUNTIME, but the call is
    // scheduled/recorded at BUILD TIME by a @BuildStep
    public RuntimeValue<DataSource> createDataSource(
            DataSourceBuildTimeConfig buildTimeConfig,
            DataSourceRuntimeConfig runtimeConfig) {

        // buildTimeConfig values are baked in at build time
        // runtimeConfig values are read from environment at startup
        AgroalDataSource ds = new AgroalDataSourceConfiguration()
            .driver(buildTimeConfig.driver())   // known at build time
            .jdbcUrl(runtimeConfig.url())        // deferred to runtime
            .build();

        return new RuntimeValue<>(ds);
    }
}
Java DatasourceBuildStep.java (build-time) 
public class DatasourceBuildStep {

    @BuildStep
    @Record(ExecutionTime.RUNTIME_INIT)
    DataSourceBuildItem setupDataSource(
            DatasourceRecorder recorder,
            DataSourceBuildTimeConfig config) {

        // This code runs at BUILD TIME
        // recorder.createDataSource() is NOT called here —
        // instead, a bytecode instruction to call it is RECORDED
        RuntimeValue<DataSource> ds =
            recorder.createDataSource(config, new RuntimeValue<>());

        return new DataSourceBuildItem(ds);
    }
}
⚠
Key insight: When you call a method on a @Recorder during a @BuildStep, Quarkus does not actually execute the method. Instead, it records the method call as bytecode that will be replayed at runtime. This is how Quarkus "pre-computes" the startup sequence.
07 The Extension Model

Three Layers

Every Quarkus extension consists of three distinct layers, each with a clear responsibility and lifecycle.

Deployment
Build-time module. Contains @BuildStep processors. Reads the Jandex index, emits build items, records runtime calls. Not included in production artifact.
Runtime
Runtime module. Contains @Recorder classes and runtime configuration. This is what ships in the production JAR/native binary. Minimal code — only what's needed at startup and request time.
SPI
Shared interface module (optional). Defines BuildItem types that other extensions can consume. Enables inter-extension communication at build time. Example: AdditionalBeanBuildItem.

Extension Maven Module Structure

Directory my-extension/ 
my-extension/
  deployment/
    pom.xml                  # Build-time dependencies (Jandex, Gizmo, etc.)
    src/main/java/
      MyExtensionProcessor.java  # @BuildStep methods
  runtime/
    pom.xml                  # Runtime dependencies only
    src/main/java/
      MyExtensionRecorder.java   # @Recorder methods
      MyExtensionConfig.java     # @ConfigMapping runtime config
  spi/                       # Optional shared BuildItem types
    pom.xml

Example BuildStep

Java MyExtensionProcessor.java 
public class MyExtensionProcessor {

    private static final String FEATURE = "my-extension";

    @BuildStep
    FeatureBuildItem feature() {
        // Registers this extension's name in the startup banner
        return new FeatureBuildItem(FEATURE);
    }

    @BuildStep
    AdditionalBeanBuildItem registerBeans() {
        // Tell Arc to discover and manage these beans
        return AdditionalBeanBuildItem.builder()
            .addBeanClasses(MyService.class, MyRepository.class)
            .setUnremovable()
            .build();
    }

    @BuildStep
    @Record(ExecutionTime.RUNTIME_INIT)
    void initializeAtRuntime(
            MyExtensionRecorder recorder,
            MyExtensionConfig config) {

        // Record a call to be replayed at runtime startup
        recorder.initialize(config.connectionString());
    }

    @BuildStep
    ReflectiveClassBuildItem registerForReflection() {
        // Required for native image: register classes that
        // must be accessible via reflection
        return new ReflectiveClassBuildItem(true, true,
            MyEntity.class.getName());
    }
}

10 Arc — CDI at Build Time

What Arc Generates

Arc is Quarkus's build-time CDI container. Unlike traditional CDI implementations (Weld, OpenWebBeans) that discover and wire beans at runtime using reflection, Arc resolves the entire bean graph at build time and generates plain Java classes for bean creation, injection, and interception.

Java Application Code 
// What you write:
@ApplicationScoped
public class OrderService {

    @Inject
    PaymentGateway gateway;

    @Inject
    InventoryService inventory;

    public Order placeOrder(OrderRequest req) {
        inventory.reserve(req.items());
        gateway.charge(req.payment());
        return new Order(req);
    }
}
Java Generated by Arc at Build Time 
// What Arc generates (simplified):
public class OrderService_Bean
    implements InjectableBean<OrderService> {

    @Override
    public OrderService create(CreationalContext<OrderService> ctx) {
        // Direct constructor call — no reflection
        OrderService instance = new OrderService();
        // Direct field set — no Field.set() reflection
        instance.gateway = arc.instance(PaymentGateway.class);
        instance.inventory = arc.instance(InventoryService.class);
        return instance;
    }

    @Override
    public Class<?> getBeanClass() {
        return OrderService.class;
    }

    @Override
    public Set<Type> getTypes() {
        // Pre-computed type set
        return Set.of(OrderService.class, Object.class);
    }
}

Why CDI Lite?

✅
Arc uses CDI Lite because it aligns with the closed world assumption. Full CDI's portable extensions rely on runtime class discovery and dynamic proxy generation, which are fundamentally incompatible with build-time optimization and GraalVM native image. CDI Lite (standardized in CDI 4.0 as "Build Compatible Extensions") was designed specifically for this use case. The trade-off: a small number of rarely-used CDI features (conversation scope, passivation, portable extensions) are not available, but 95%+ of real-world CDI usage works identically.

Key Themes

1. JPMS (Java Module System)

Quarkus 4.0 targets modular runtime architecture using the Java Platform Module System. Extensions will declare explicit module boundaries, enabling stronger encapsulation, smaller native images, and better startup via jlink custom runtime images. This is the most architecturally significant change since the augmentation model.

2. Java 21+ Baseline

Java 21 (LTS) becomes the minimum required version. This unlocks first-class virtual threads (Project Loom), structured concurrency, pattern matching, record patterns, sequenced collections, and scoped values. Virtual threads will become the default execution model for blocking workloads, replacing the traditional worker thread pool.

3. AI-Native Development

Deep integration with LangChain4j, model-agnostic AI service abstraction, built-in RAG pipelines, and AI dev services (local Ollama containers). The quarkus-langchain4j extension family becomes a first-class pillar alongside REST, messaging, and data. Guardrails, prompt management, and AI observability baked in.

4. Project Leyden & IBM Semeru

Integration with OpenJDK Project Leyden (ahead-of-time class initialization, training runs) and IBM Semeru Cloud Compiler. Goal: JVM-mode startup times approaching native image (<100ms) with full JIT throughput. CRaC (Coordinated Restore at Checkpoint) support for instant restore from snapshots.

5. Spring Compatibility Layer

Expanded Spring API compatibility to ease migration. Quarkus 4.0 targets support for Spring Web, Spring Data JPA, Spring Security, Spring Cloud Config, and Spring Scheduling annotations — processed at build time via the Quarkus augmentation model rather than runtime reflection.

6. JMS + CLI Enhancements

Jakarta Messaging (JMS) 3.1 support via a new reactive connector. Quarkus CLI enhancements: project templates, extension scaffolding, deployment commands. The CLI becomes the primary developer entry point, replacing Maven archetypes for project creation.
