01 Top-Down Layer View

Click any layer to expand it and see what lives inside. Every Quarkus request travels through all five layers — understanding each one tells you where your CPU time, memory, and I/O go.

Java 21 CDI Beans JAX-RS
▼
Bean Lifecycle

Arc manages bean creation/destruction. @ApplicationScoped beans live for the app lifetime. @RequestScoped beans are created per HTTP request and destroyed after the response.

Memory Footprint

Your code compiles to bytecode loaded into Metaspace. Bean instances live on the heap. Each CDI proxy is a small synthetic class — also Metaspace.

Build-Time Wiring

Arc resolves all @Inject points at build time. The runtime only loads pre-generated wiring classes — zero reflection, zero classpath scanning.

Thread Model

Non-blocking endpoints (Uni/Multi) run on the Vert.x event loop. @Blocking methods offload to the Quarkus worker thread pool.

⚡
Quarkus Runtime
Arc (CDI) · RESTEasy Reactive · SmallRye · Vert.x · Netty
Vert.x Netty Arc
▼
Vert.x Event Loop

Quarkus uses Vert.x as its reactive engine. The event loop runs on N threads (usually 2× CPU cores). Each thread handles thousands of concurrent connections — no thread-per-request overhead.

Netty I/O

Vert.x sits on top of Netty for NIO. Netty uses direct (off-heap) ByteBuf buffers for HTTP parsing and I/O, keeping pressure off the GC heap.

RESTEasy Reactive

Route table is pre-computed at build time. Request dispatch is a direct map lookup — O(1). No dynamic proxy chains. Filters run inline on the event loop.

Worker Pool

Blocking operations use a separate worker thread pool (default: 200 threads, configurable). Workers hand results back to the event loop via Uni/Multi.

🖥️
JVM (OpenJDK / GraalVM)
Heap · Metaspace · Code Cache · GC · JIT Compiler · Thread Stacks
G1GC JIT NMT
▼
Heap

All object allocations. Quarkus default: -Xms256m -Xmx512m in containers. Tuned lower than Spring Boot because far fewer objects are created at startup (build-time wiring means no framework object graphs).

Metaspace

Loaded class metadata. Quarkus loads significantly fewer classes than Spring Boot — typically 3–4× fewer. Default metaspace cap: 256m. No PermGen (removed in JDK 8+).

Code Cache

JIT-compiled native code. Quarkus hot paths get JIT-compiled quickly. Default: 240m. Tiered compilation (C1 → C2) means peak throughput is reached after ~60s of warm traffic.

Thread Stacks

Each thread uses -Xss512k (default) of native memory. Vert.x event loop: ~2–4 threads. Worker pool: up to 200. Total stack memory: ~100MB at full load — off-heap, not GC-managed.

🐧
Operating System (Linux / UBI)
Virtual memory · page cache · file descriptors · cgroups · OOM killer
cgroups v2 mmap OOM
▼
RSS vs VSZ

RSS (Resident Set Size) = physical RAM in use. VSZ = virtual address space mapped. Quarkus JVM RSS is typically 200–400MB. VSZ is much higher due to memory-mapped JARs and shared libs.

Container Limits

Kubernetes sets limits.memory enforced by cgroups. JVM reads container limits via -XX:+UseContainerSupport (default on JDK 11+). Heap defaults to 25% of container memory limit.

File Descriptors

Each open socket, file, and pipe uses a file descriptor. Vert.x NIO servers need higher ulimits. Default OS limit is 1024 — Quarkus recommends 65536 for production workloads.

OOM Killer

If the container exceeds its memory limit, the Linux OOM killer terminates the process. Set JVM heap ceiling below the cgroup limit: if limit is 512MB, use -Xmx384m to leave headroom for Metaspace and stacks.

⚙️
Hardware / Container Runtime
CPU cores · NUMA nodes · physical RAM · NVMe disk · network interfaces
CPU Affinity NUMA IOPS
▼
CPU Throttling

Kubernetes limits.cpu is enforced via cgroup CPU quotas. JVM detects CPU count incorrectly if not using +UseContainerSupport. G1GC sizes its parallel workers based on available CPUs.

NUMA Awareness

Multi-socket servers have NUMA nodes — accessing remote memory is ~40% slower. G1GC is NUMA-aware. Quarkus native binaries benefit from NUMA placement since the binary is a single process.

Disk I/O (JAR loading)

At JVM startup, the JDK reads class files from JAR. Quarkus fast-jar format uses an index so only needed classes are loaded. Cold-start on slow NFS/EFS mounts can add 2–5s of I/O wait.

Network

Vert.x NIO sockets use kernel bypass where available. In Kubernetes, pod-to-pod traffic goes through CNI plugins (Cilium, Calico). Each hop adds ~0.1ms latency — avoid unnecessary service mesh hops on hot paths.

04 JVM Flags & Tuning

Quarkus sets sensible defaults. These are the flags you'll need to override for production banking workloads — each one is a deliberate trade-off between memory, CPU, and latency.

Heap Sizing
-Xms256m
Initial heap. Set equal to Xmx to avoid heap resize pauses in production.
-Xmx512m
Max heap. Leave 128–256MB headroom below container memory limit for Metaspace + stacks.
-XX:+UseContainerSupporttrue
On by default JDK 11+. Makes JVM read cgroup memory limits correctly in Kubernetes.
-XX:MaxRAMPercentage75.0
Alternative to Xmx — sets heap as % of container memory. Better for dynamic pod sizing.
GC Tuning (G1GC)
-XX:+UseG1GCdefault
Default collector JDK 11+. Handles heap sizes from 256MB to large heaps gracefully.
-XX:MaxGCPauseMillis20
Target max GC pause in ms. G1 adjusts region sizes to try to hit this. Set to 50ms for higher throughput.
-XX:G1HeapRegionSize4m
Larger regions for larger heaps. Default auto-selected. Set explicitly if you see humongous allocation warnings.
-XX:+ParallelRefProcEnabledtrue
Process reference objects in parallel during GC. Reduces pause time on high-allocation workloads.
Metaspace & Code Cache
-XX:MetaspaceSize128m
Initial Metaspace size. Set equal to MaxMetaspaceSize to avoid class metadata GC on startup.
-XX:MaxMetaspaceSize256m
Cap for class metadata. Quarkus uses ~80–100MB. Set a ceiling to avoid unconstrained growth.
-XX:ReservedCodeCacheSize128m
JIT code cache. Default 240MB is too large for constrained containers. 128MB is sufficient for Quarkus.
-XX:+TieredCompilationtrue
C1 (fast compile) then C2 (optimised). Enabled by default. Gives fast startup + peak throughput after warmup.
Thread Stack & IO
-Xss512k
Thread stack size. Default is 512K–1M depending on OS. Reduce to 256K for services with very high worker thread counts.
-Djava.net.preferIPv4Stacktrue
Avoids IPv6 resolution delays in some Kubernetes CNI environments. Common fix for connection timeout issues.
-Dfile.encodingUTF-8
Always set explicitly in containers. Default encoding depends on OS locale which may not be UTF-8 in minimal images.
-Dvertx.disableDnsResolvertrue
Use JDK DNS resolver instead of Vert.x async resolver. Needed in some Kubernetes environments where Vert.x DNS misbehaves.

Regulated banking services: For high-throughput payment APIs, set -Xms equal to -Xmx to prevent heap resizing under peak load. Use -XX:MaxGCPauseMillis=10 to keep P99 latency below 50ms. For long-running batch jobs, allow larger heap and use -XX:MaxGCPauseMillis=200 to favour throughput over pause time.

Decision rule for regulated services: Use JVM mode for high-throughput payment APIs, account services, and anything requiring JFR/heap dump diagnostics in production.
